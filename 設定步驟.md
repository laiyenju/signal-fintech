# SIGNAL 每 3 小時自動更新：設定步驟（Claude Code 雲端排程）

這份指南帶你把 SIGNAL 設定成「每 3 小時自動更新、電腦可關機、不花按次計費 API 錢」。
整個機制的分工是：

```
Claude Code 雲端排程（每 3 小時醒來，跑在 Anthropic 雲端）
    → 執行 scripts/fetch_news.py 抓 RSS（純抓取，不呼叫 AI）
    → （可選）用 Readwise connector 撈過去 48h 訂閱內容當額外候選
    → Claude Code 自己讀規則選稿＋兩段式改寫（用你的訂閱額度，非計費 API）
    → 產出 data.json，從 claude/* 分支開 PR 並合併回 main
GitHub Pages（免費靜態網站託管）
    → 自動部署 main
使用者打開網站
    → index.html 開站時 fetch 同目錄的 data.json → 顯示最新內容
```

完整選稿／改寫／提交規則見 [`排程任務指令.md`](./排程任務指令.md)。英文產品說明見 [`README.md`](./README.md)。

---

## 你會用到的檔案

| 檔案 | 放在 repo 的位置 | 用途 |
|---|---|---|
| `index.html` | 根目錄 | 網站本體，開站時抓 `data.json` |
| `data.json` | 根目錄 | 內容資料，排程依規則更新（非整檔無關覆寫 cover） |
| `scripts/fetch_news.py` | `scripts/` | 抓 RSS，產出 `raw_items.json`（純抓取、不呼叫 AI、零成本） |
| `排程任務指令.md` | 任意（給你複製貼上用） | 貼進 Claude Code 排程任務的指令內容 |

> 選稿改寫由 Claude Code 排程任務自己做，**不需要**計費 API 腳本或 GitHub Actions workflow。

---

## 步驟一：建立 GitHub repo 並放入檔案

1. 在 GitHub 建一個新的 repo（例如 `signal-fintech`），可設為 Public（GitHub Pages 免費方案需要 Public，除非你有付費方案支援 Private Pages）。
2. 把以下檔案放進去：
   - 根目錄：`index.html`、`data.json`
   - `scripts/` 資料夾：`fetch_news.py`
3. `scripts/fetch_news.py` 的 `FEEDS` 清單已附上所有來源的實際 feed 網址，不需要額外填寫；個別來源若抓取失敗，腳本會自動略過並印出錯誤，不影響其他來源。

---

## 步驟二：開啟 GitHub Pages

1. 到 repo 的 **Settings → Pages**。
2. Source 選 **Deploy from a branch**，Branch 選 `main`、資料夾選 `/ (root)`，儲存。
3. 稍等一兩分鐘，GitHub 會給你一個網址（形如 `https://<你的帳號>.github.io/signal-fintech/`）。
4. 打開該網址，確認網站能正常顯示初始的 `data.json` 內容（封面、本週要聞、來源清單都出得來）。

> 因為 `index.html` 是抓「同目錄」的 `data.json`，只要兩個檔案都在 repo 根目錄、由同一個 Pages 部署，就不會有跨網域問題。

---

## 步驟三：設定 Claude Code 雲端排程任務

1. 進入 Claude Code 的雲端排程功能（Scheduled Tasks / Routines）。
2. 建立一個新的排程任務，指向你剛才建立的 GitHub repo（讓任務有權限 clone、修改、push 這個 repo；提交流程會用 `gh pr create` / `gh pr merge`）。
3. 排程頻率設為**每 3 小時**（cron 例如 `0 */3 * * *`）。若要更即時可改每小時 `0 * * * *`，但會多耗訂閱額度；RSS 變動通常不快，預設 3 小時已足夠。
4. 把 `排程任務指令.md` 的**全部內容**貼進這個任務的指令欄位，作為它每次執行要做的事。
5. （可選）若環境有 Readwise connector，排程會多撈 48 小時 Reader 訂閱當候選；沒有或失敗時任務仍會繼續，只靠 RSS。
6. 儲存並啟用。

> Claude Code 排程跑在 Anthropic 雲端，所以**你的電腦可以關機**，不影響每 3 小時執行。
> 實際的介面名稱、所屬訂閱層級（Pro／Max／Team／Enterprise）與可用性可能會變動，
> 若設定畫面與此處描述不同，以 Claude Code 官方文件的當前說明為準。

---

## 步驟四：驗證

1. 在排程任務設定完後，先手動觸發一次（多數排程工具都有「立即執行一次」的選項）。
2. 執行完到 GitHub repo 看是否有新的 PR／merge、`data.json` 的 `_generated_at` 是否更新。
3. 重新整理網站，確認內容有換成最新、頁尾「內容更新於…」時間有更新。
4. 打開 repo 的 **Wiki**，確認當日有 `<YYYY-MM-DD>` 日誌頁（或當日頁多了一筆 run）；任務輸出／PR body 應含 `newsroom_wiki=ok`（失敗則為 `failed`，見下方連通說明）。
5. 確認無誤後，接下來它就會每 3 小時自己跑，不需要你介入。

## Wiki 選稿日誌連通（階段 0 診斷紀錄）

選稿日誌寫在獨立 git repo：`laiyenju/signal-fintech.wiki`（網頁 Wiki）。  
與 `data.json` 解耦後，**主 repo 推送成功 ≠ wiki 有寫入**。2026-07-19 改推 wiki 後，站點持續更新但 wiki 停更，即此斷點。

### 決策（已拍板）

| 項 | 選擇 |
|----|------|
| wiki 寫入失敗 | **Soft-fail**：不阻擋 `data.json` 出刊，但任務／PR 必標 `newsroom_wiki=failed` |
| 認證寫法 | **文件以 `gh`／HTTPS 為主路徑**（貼近 Claude Code + `gh` 環境）；SSH 作備援 |

### 本機診斷結果（2026-07-24，帳號 `laiyenju`）

| 路徑 | clone | push（dry-run） |
|------|-------|-----------------|
| SSH `git@github.com:…/signal-fintech.wiki.git` | OK | OK |
| `gh repo clone laiyenju/signal-fintech.wiki`（HTTPS） | OK | OK |
| `https://x-access-token:$(gh auth token)@github.com/…wiki.git` | OK | OK |

其他：

- repo `has_wiki: true`；目前使用者對主 repo 為 `admin` + `push`
- 本機 `gh`：HTTPS protocol、token scope 含 `repo`（使用者 token 通常可寫 wiki）
- 本機主 repo remote 為 SSH；API 列出的 **Deploy keys 為空**（本機能 push wiki 靠的是**使用者金鑰／gh 登入**，不是 repo deploy key）
- 本機另有名為 `signal-push-20260718` 的金鑰（`~/.ssh/id_signal`）——若排程只掛「僅限主 repo」的 deploy key，**常常不能 push `*.wiki.git`**（wiki 是另一個 repo）

### 重要限制：本機通過 ≠ 雲端排程通過

排程跑在 **Claude Code／Anthropic 雲端**，憑證與本機不同。階段 0 **無法在此環境代替 Routine 實測**。  
下一輪手動觸發排程後，請在任務 log 搜尋：

- `newsroom_wiki=ok` → 雲端寫入已通  
- `newsroom_wiki=failed reason=clone|push|…` → 依 reason 補權限  
- **完全沒有** `newsroom_wiki=` 字樣 → agent 可能跳過 13.6，需更新貼上的 `排程任務指令.md`

### 建議在「排程同一環境」貼上的連通檢查（可手動跑一次）

```bash
# 優先：與 gh pr 同一套登入
rm -rf /tmp/newsroom-wiki-test
gh repo clone laiyenju/signal-fintech.wiki /tmp/newsroom-wiki-test -- --depth 1 \
  && git -C /tmp/newsroom-wiki-test push --dry-run origin HEAD:master \
  && echo "WIKI_GH_OK"

# 備援：SSH（僅當環境有可用的 git@github.com 金鑰）
rm -rf /tmp/newsroom-wiki-ssh
git clone --depth 1 git@github.com:laiyenju/signal-fintech.wiki.git /tmp/newsroom-wiki-ssh \
  && git -C /tmp/newsroom-wiki-ssh push --dry-run origin HEAD:master \
  && echo "WIKI_SSH_OK"
```

若 `gh` 可 clone 主 repo、卻不能 clone wiki：到 GitHub 確認 Wiki 已啟用，且排程用的身分對該帳號／repo 有寫入權（使用者 PAT/`repo` scope，或可寫 wiki 的機制；**單 repo deploy key 通常不夠**）。

### 排程 13.6（已寫入 `排程任務指令.md`）

1. **主路徑**：`gh repo clone laiyenju/signal-fintech.wiki` → `NEWSROOM_DIR=…` 跑 `newsroom.py` → `git add/commit/push`  
2. **備援**：SSH clone（主路徑失敗時再試）  
3. 兩路皆失敗 → `newsroom_wiki=failed`（soft-fail，不擋 data.json）  
4. 成功 → `newsroom_wiki=ok`；published 時 PR body 與任務收尾（第 15 節）必含狀態行  
5. **請把更新後的 `排程任務指令.md` 全文重新貼進 Claude Code 排程任務**（只改 repo 不會自動改雲端貼上的舊指令）

## 常見狀況

- **網站顯示「暫時無法載入最新內容」**：多半是 `data.json` 不在 index.html 同目錄，或 JSON 格式壞掉。檢查兩個檔案是否都在 repo 根目錄、`data.json` 是否為合法 JSON。
- **某些來源一直抓不到**：先確認 `fetch_news.py` 裡該來源的 feed 網址是否還有效（來源網站偶爾會改版換網址）。個別來源失敗不會影響其他來源。
- **台灣長期都是「本日觀察」**：代表台灣來源當天沒有夠格的重大新聞，這是設計上的誠實表現；若希望台灣內容更即時，需要再擴充中文來源（可再回來找我盤點 RSS）。
- **PR 已開但沒自動合併**：排程指令會盡力 `gh pr merge`；若權限不足，需到 GitHub 手動 merge 一次，並檢查 routine 的 repo 權限。
- **Wiki 只有 7/18、7/19、之後都沒日誌**：主站可能仍在更新；是 13.6 寫 wiki 失敗或被跳過。看任務輸出的 `newsroom_wiki=`，並跑上方連通檢查。
- **想調整選稿嚴格度或改寫風格**：直接編輯 `排程任務指令.md` 的規則段落，再更新排程任務的指令欄位即可，網站端不用動。

## 之後若要擴充來源

新增來源時，兩個地方要一起更新，維持一致：
1. `scripts/fetch_news.py` 的 `FEEDS` 清單（加抓取來源）。
2. `index.html` 的 `SOURCES` 物件（讓「追蹤來源清單」顯示出來）。

`SOURCES` 只應列出**實際有在抓或有在配對**的來源；已停用的 feed（例如壞掉的 Bloomberg RSS）不要留在清單裡。
