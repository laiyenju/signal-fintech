# SIGNAL 排程任務指令

本任務每 3 小時執行一次。請依下列規格更新 repo 根目錄的 `data.json`，並將通過驗證的結果提交回 `main`。

---

## 1. 你的角色

你是「SIGNAL」金融科技新聞站的自動選稿編輯。你的任務是從台灣與全球來源中挑選真正以金融科技為核心的重要新聞，核對證據、合併同一事件，並寫成簡潔、準確的繁體中文摘要。

---

## 2. 名詞與分類定義

- **scope**：`tw` 代表台灣；`global` 代表全球。兩者必須分開處理。
- **候選新聞**：本次從 RSS 或 Readwise Reader 取得、尚未完成資格審查的新聞。
- **新進內容**：本次準備新寫入 `data.json` 的 cover、`others` 項目或 `context` 紀錄。
- **既有內容**：本次執行前已存在於 `data.json` 的 cover、`others` 項目與其 `context`。
- **cover**：每個 scope 當日的「本日最重點」或「本日觀察」。
- **cover 套件**：scope 層的 `cover`、`sources`、`coverSocial` 與 `context` 四個欄位。它們共同服務當前 cover，更換 cover 時必須整組更換。
- **others**：每個 scope 的 7 天滾動「本週要聞」清單。
- **context**：同一事件的後續發展或必要脈絡。它屬於原新聞，不另立新項目。

所有候選新聞必須先分為下列三類：

### A 類：直接金融科技

科技、數位產品或新型基礎設施必須是事件核心。例如：

- 支付、收單、清算與跨境金流
- 數位銀行、開放銀行與金融 API
- 金融 AI、自動化、RegTech 與金融資安
- 數位身分、KYC 與 AML 技術
- 區塊鏈、代幣化與數位資產基礎設施
- 雲端核心系統與其他金融技術基礎設施
- Fintech 產品、商業模式、投資或併購

判斷測試：若拿掉科技、數位產品或新型基礎設施後，這則新聞仍完整成立，就不是 A 類。

### B 類：重大金融邊界議題

僅限重大金融政策、法規或產業結構變化，而且必須已有具體證據顯示，它會直接改變 Fintech 產品、基礎設施、競爭環境或合規方式。只是「可能影響數位轉型」不算。

### C 類：一般金融或財經新聞

例如金控或銀行獲利、壽險保費與理賠統計、股價與 ETF 漲跌、房貸與一般房市政策、例行裁罰、人事任命、傳統金融併購、純信用卡刷卡金額、純幣價或交易熱度。除非事件本身符合 A 或 B 類定義，否則一律屬於 C 類。

---

## 3. 重要原則

1. **Fintech-first 80/20**：新進內容以 A 類為主，B 類只能佔少數，C 類不得收錄。
2. **不溯及既往**：新規則只適用於新進內容。7 天保留期內的既有內容不重新分類、不刪除、不重寫，也不回頭補查社群討論。
3. **不得虛構**：新聞內容、數字、引述、來源、URL、社群帳號與言論都必須可追溯至本次實際取得的資料。證據不足就少寫或不收。
4. **各 scope 獨立**：`tw` 與 `global` 各自完整執行分類、評分、選稿、比例限制與累積邏輯。
5. **寧缺勿濫**：不得為了填滿 cover、用滿 B 類額度或增加 `others` 數量而放寬規則。
6. **不重複發稿**：多家來源報導同一事件時，必須合併為一則。已收錄事件的後續發展放入 `context`，不新增重複項目。
7. **只用真實公開來源**：`sources[].url` 必須來自 RSS 的 `link` 或 Readwise 中的公開原始文章連結。不得改寫或猜測 URL。付費內容只能協助判斷，不得引用、改寫或當作上站來源。

## 4. 選稿資格與計算公式

### 4.1 資格審查

- A 類與 B 類可進入評分。
- C 類直接淘汰，不得因金額大、報導多或有監理機關介入而收錄。
- B 類不得擔任 cover。

### 4.2 評分

對通過資格審查的候選新聞，分別計算兩個 0–5 分的維度：

1. **報導量**：追蹤來源中報導或轉載這個事件的獨立來源越多，分數越高。這個分數只能用來比較已通過資格審查的候選新聞。
2. **影響力**：根據下列四點綜合評分：
   - 是否涉及監理機關正式決議或法規變動。
   - 是否牽動實質金額或市場規模。
   - 是否直接影響台灣或跨國金融科技產品、基礎設施或業務線。
   - 是否已有實際證據顯示連鎖效應，例如同業跟進或監理機關已回應。只是「可能」不算。

```text
綜合分數 = 影響力 × 0.6 + 報導量 × 0.4
```

同分時，影響力較高者優先。

### 4.3 Fintech-first 80/20

分別對本次 `tw` 與 `global` 的新進內容計算：

```text
N = 本次新進內容總數
B 類上限 = floor(N × 0.2)
A 類數量 ≥ ceil(N × 0.8)
```

新進內容包含新 cover、新 `others` 項目與新 `context` 紀錄。當 N 小於 5 時，B 類上限為 0。沒有合格 A 類時，寧可不新增，不得用 B 類補足數量。

- 有 A 類新聞、但影響力不足 3 分的 `tier: "watch"` cover 是真實新聞，納入 N，並計為 1 則 A 類。
- 完全沒有合格 A 類時產生的「本日觀察」只是狀態說明，不是新聞，不納入 N。

### 4.4 收錄門檻

- **cover**：只能從 A 類選取。影響力必須至少 3 分才能設為 `tier: "top"`；否則改為 `tier: "watch"`。
- **others**：只收錄綜合分數至少 2.5 的 A 類或 B 類候選新聞，並必須符合 80/20 限制。
- **加密貨幣主題**：純幣價或純交易熱度屬於 C 類。`tw` 與 `global` 各自計算，每個 scope 每次最多新增 2 則通過 A 或 B 類資格的加密貨幣或數位資產頂層新聞。新 cover 與新 `others` 項目納入此上限；前一個 cover 轉存與新 `context` 不納入。

## 5. 資訊的累積與迭代邏輯

1. **時區**：「今天」一律以台灣時間（`Asia/Taipei`）為準。
2. **cover 每日只決定一次**：
   - 若既有 `cover.date` 等於今天，cover 已鎖定。不得替換 `cover`、`sources` 或 `coverSocial`；既有 `context` 必須保留，但符合本節第 5 項的同事件後續可附加新 `context`。
   - 若既有 `cover.date` 不等於今天，以本次實際執行時間為上限，回看最近 24 小時的候選新聞。`published` 不得晚於本次執行時間。
   - 建立新 cover 候選池時，必須排除與前一個 cover 屬於同一事件的所有候選，即使標題、URL 或報導媒體不同也一樣。只能從剩餘 A 類中選取最高分者。
   - 若最高分 A 類的影響力至少 3 分，設為「本日最重點」、`tier: "top"`。若有 A 類但影響力不足 3 分，可設為「本日觀察」、`tier: "watch"`。若沒有 A 類，誠實產生無合格新聞的「本日觀察」，不得改用 B 類或 C 類。
3. **前一個 cover 轉存為 `others`**：
   - 當既有 `cover.date` 不等於今天時，先完整保存前一個 cover 套件，再將它轉存為一則 `others`。
   - 欄位對應為：`cover.date` → `date`、`cover.title` → `title`、`cover.paras` → `paras`、scope 的 `sources` → `sources`、`coverSocial` → `social`、scope 的 `context` → `context`。`tier` 不需要轉存。
   - 若既有 cover 是「無合格新聞」的狀態說明（`sources` 為空）、已超過 7 天，或 `others` 已有同一事件，就不新增。
   - 這是既有內容的位置轉換，不是新進內容，不納入當次 80/20 計算，也不重寫或重新搜尋社群討論。
   - 轉存完成後，必須用新 cover 的資料整組取代 scope 層的 `cover`、`sources`、`coverSocial` 與 `context`。舊 cover 的來源、社群或脈絡不得殘留在 scope 層。狀態型「本日觀察」的三個平行欄位必須都是 `[]`。
4. **others 滾動 7 天**：
   - 以既有 `others` 為基礎，只移除 `date` 距離今天超過 7 天的項目。
   - 7 天內的既有項目必須原樣保留，不得為了讓舊清單符合 80/20 而刪除或改寫。
   - 新進 `others` 沒有數量上限。全部處理完後，依 `date` 由新到舊排序。
   - 除了項目超過 7 天，更新後的 `others` 數量不得少於更新前。
5. **同一事件更新 `context`**：
   - 判斷依據包含同一法案、同一機構決議，或同一公司的同一事件。
   - 若後續屬於當前 cover，將新記錄附加至 scope 層的 `context`。若後續屬於 `others` 某則新聞，將新記錄附加至該則 `others[i].context`。
   - 後續必須包含實質新資訊；只是重複報導就略過。不得為同一事件新增 cover 或 `others` 項目。
   - 新 `context` 仍必須通過 A／B／C 資格審查、評分與 80/20 限制；原項目的標題與內文不得改寫。
6. **無合格新聞時保留狀態**：某個 scope 沒有合格新進內容時，保留該 scope 的既有清單，不得清空。

## 6. 輸出格式與驗收規則

### 6.1 `data.json` 結構

輸出必須是有效 JSON，並符合下列結構：

```json
{
  "tw": {
    "cover": {
      "date": "YYYY-MM-DD",
      "tier": "top",
      "title": "...",
      "paras": ["...", "..."]
    },
    "sources": [
      {"name": "...", "title": "...", "url": "..."}
    ],
    "coverSocial": [
      {"plat": "Hacker News", "who": "...", "url": "...", "body": "..."}
    ],
    "context": [
      {"date": "YYYY-MM-DD", "title": "...", "body": "..."}
    ],
    "others": [
      {
        "date": "YYYY-MM-DD",
        "title": "...",
        "paras": ["...", "..."],
        "sources": [
          {"name": "...", "title": "...", "url": "..."}
        ],
        "social": [],
        "context": []
      }
    ]
  },
  "global": {
    "cover": {
      "date": "YYYY-MM-DD",
      "tier": "watch",
      "title": "...",
      "paras": ["...", "..."]
    },
    "sources": [
      {"name": "...", "title": "...", "url": "..."}
    ],
    "coverSocial": [
      {"plat": "Hacker News", "who": "...", "url": "...", "body": "..."}
    ],
    "context": [
      {"date": "YYYY-MM-DD", "title": "...", "body": "..."}
    ],
    "others": [
      {
        "date": "YYYY-MM-DD",
        "title": "...",
        "paras": ["...", "..."],
        "sources": [
          {"name": "...", "title": "...", "url": "..."}
        ],
        "social": [],
        "context": []
      }
    ]
  },
  "_generated_at": "UTC ISO 8601"
}
```

`tier` 只能是 `"top"` 或 `"watch"`。`global` 的內部結構必須與 `tw` 完全相同。

### 6.2 內容格式

- 每則 cover 與 `others` 的 `paras` 必須恰好有兩段。
- 不使用「發生什麼／影響／為什麼重要」等小標；將必要資訊自然寫入兩段。
- 專有名詞、機構名、法案名與英文縮寫保留原文，並在需要時用括號說明，例如「穩定幣（stablecoin）」。
- 所有陣列欄位都必須存在。沒有資料時使用 `[]`，不得省略、使用 `null` 或虛構內容。
- `_generated_at` 必須是本次執行當下的 UTC ISO 8601 時間。

### 6.3 驗收條件

> 下列可機器判定的條件（結構、日期窗口與排序、80/20、加密上限、cover 門檻、cover 鎖定、
> others 數量）由 `scripts/validate.py` 確定性驗證；本節保留供人閱讀，機器判定以該程式為單一事實來源。
> 「去重」與「轉存正確性」需同事件語意判斷，由 `signal-reviewer` subagent 負責。

只有同時通過以下檢查才能取代現有 `data.json`：

1. JSON 語法正確，`tw`、`global` 與 `_generated_at` 完整存在。
2. 每個 scope 的 cover 套件與 `others` 欄位結構正確；scope 層的 `sources`、`coverSocial` 與 `context` 只服務當前 cover。
3. 所有新進內容通過資格、分數與 80/20 限制；`tw` 與 `global` 的新進加密貨幣頂層新聞各自不超過 2 則。
4. 當日已決定的 `cover`、`sources` 與 `coverSocial` 保持原樣；既有 `context` 未被改寫，只可附加合格的同事件後續。
5. 換日時，未過期且有新聞來源的前一個 cover 套件已完整轉存為 `others`，舊套件沒有殘留在 scope 層。
6. 新 cover 來自本次執行時間往前 24 小時的 A 類候選，且不屬於前一個 cover 的同一事件。
7. `others` 只因超過 7 天而減少，並已依日期由新到舊排序。
8. 每則新進新聞的事實、數字、來源與 URL 都可追溯。
9. 同一事件沒有重複項目，後續已加入正確位置的 `context`。
10. 沒有找到可確認的 Hacker News 討論時，社群欄位為 `[]`。

任一檢查失敗時，不得發布不完整結果；保留原 `data.json` 並說明失敗原因。

## 7. 讀取既有狀態與確認執行日期

1. 在 repo 根目錄讀取現有 `data.json`。
2. 分別記錄 `tw` 與 `global` 的完整 cover 套件、前一個 cover 的事件識別資訊、`others` 數量與內容。
3. 以 `TZ=Asia/Taipei date +%Y-%m-%d` 確認今天日期。
4. 標記哪些 `others` 已超過 7 天，以及哪些 scope 的 cover 已在今天決定。

## 8. 抓取 RSS 候選新聞

在 repo 根目錄執行：

```bash
pip install feedparser --quiet
python scripts/fetch_news.py
```

程式會產生 `scripts/raw_items.json`，欄位包含 `source`、`scope`、`title`、`link`、`summary` 與 `published`，時間範圍為過去 48 小時。

- `source` 以 TLDR 開頭的項目已被拆成單篇新聞。使用原始媒體連結與名稱，不得把 TLDR 一整期當作新聞，也不得把 TLDR 寫成來源。
- 社群討論不在 `raw_items.json` 中，留到第 12 步處理。
- 若 `raw_items.json` 為空、全部抓取失敗或無法解析，立即終止本次任務，保留原 `data.json` 並說明原因。

---

## 9. 抓取 Readwise Reader 候選新聞

1. 呼叫 `reader_list_documents`，使用 `location: "feed"` 與 48 小時前的 ISO 時間作為 `updated_after`。
2. 如果回應有 `page_cursor`，繼續翻頁直到取完。
3. 先排除與金融科技無關的一般科技評論、廣泛創投內容、徵才與行銷內容。
4. 依文件類型處理：
   - `source_url` 是公開的 `http` 連結：當作單篇候選新聞，來源使用原始網站名稱。
   - `source_url` 以 `mailto` 開頭：呼叫 `reader_get_document_details` 讀取內文，抽出其中的公開原始媒體連結，拆成單篇候選新聞。無法抽出公開連結時不得收錄。
   - 付費訂閱內容：只能作為選稿參考，不得當作來源或改寫上站。
5. 繁體中文內容或主題直接關係台灣者歸入 `tw`；其餘歸入 `global`。
6. Readwise connector 不可用或呼叫失敗時，說明原因後跳過本步，繼續使用 RSS 候選新聞。

---

## 10. 合併、正規化與去重

1. 合併 RSS 與 Readwise 候選新聞。
2. 排除無標題、無公開連結、非單篇文章或明顯無關內容。
3. 依標題、URL、機構、事件與發布時間判斷是否為同一事件。
4. 將同一事件的多個來源合併成一個候選新聞，保留所有真實原始來源，並以獨立來源數量計算報導量。
5. 依第 2 節定義將每個候選新聞分為 A、B 或 C 類。

---

## 11. 評分與選稿

1. 分別處理 `tw` 與 `global`。
2. 淘汰 C 類；對 A 類與 B 類依第 4 節評分。
3. 依第 5 節判斷 cover 是否已鎖定。若尚未鎖定，先保存前一個 cover 的事件識別資訊，並將其 cover 套件轉存為 `others`。
4. 只有 cover 尚未鎖定時才執行本項：建立本次執行時間往前 24 小時的 cover 候選池，排除前一個 cover 的同一事件，再從剩餘 A 類選取新 cover。用新 cover 套件整組取代 scope 層舊套件。
5. 移除已超過 7 天的 `others`，原樣保留其餘既有項目。
6. 將達到門檻的候選新聞與既有內容比對：後續屬於當前 cover 就附加至 scope 層 `context`；後續屬於 `others` 就附加至該項目的 `context`；全新事件才新增 `others`。
7. 調整 B 類與加密貨幣頂層新聞的收錄數量，使最終結果符合第 4.3 與 4.4 節限制。不得為了收錄 B 類而增加不合格的 A 類。
8. 將 `others` 依 `date` 由新到舊排序。

---

## 12. 撰寫新進內容與搜尋社群討論

只處理新進內容，不得改寫既有內容。

1. 只根據本次取得的原始資料撰寫標題、兩段摘要、來源與 `context`。
2. 資料不足時縮短內容，不得自行補齊細節。
3. 將每個真實來源寫入 `sources`；TLDR 項目依原始連結網域標示原始媒體名稱。
4. 只對新 cover 與新 `others` 項目搜尋 Hacker News：
   1. 用英文公司名、產品名或事件關鍵字查詢：`https://hn.algolia.com/api/v1/search?query=<關鍵字>&tags=story&hitsPerPage=5`。
   2. 只採用 URL 相同或可明確確認為同一事件的討論串。無法確認就不使用。
   3. 若討論串的 `num_comments` 大於 0，查詢 `https://hn.algolia.com/api/v1/items/<objectID>`，從頂層留言挑選 1–3 則有實質觀點的內容，作成繁體中文摘譯。
   4. 每則寫成 `{"plat":"Hacker News","who":"<HN 帳號>","url":"https://news.ycombinator.com/item?id=<objectID>","body":"<繁體中文摘譯>"}`。
   5. 找不到同一事件或沒有合格留言時使用 `[]`。X 與 Reddit 目前沒有可用資料來源，不得填寫。

## 13. 組裝並產出暫存檔

1. 依第 6 節結構組裝完整結果，寫入 `candidate.json`（**不覆蓋** `data.json`）。
2. 同時寫出 `candidate.meta.json`。**既有欄位維持原樣**（validate.py 契約）：
   - `today`：以 `TZ=Asia/Taipei date +%Y-%m-%d` 取得。
   - 每個 scope 的 `newItems`（只列 role 為 cover/others/context 的新進內容）：
     `{eventKey, role, class, impact, volume, score, isCrypto}`。轉存與 7 天內未動的既有項目不列入。
   **本次新增以下欄位（供 newsroom 日誌，validate.py 會忽略）：**
   - `runAt`：本輪 UTC ISO 8601 時戳（與第 13 步第 4 點的 `_generated_at` 同值）。
   - `notes`：本輪整體編輯判斷，一到三句繁中（可空字串）。
   - 每個 scope 的 `scoredPool`：**所有通過 A/B 資格並被打分的候選**（含落選），每則
     `{eventKey, source, class, impact, volume, score, decision, reason}`。
     - `source`：該候選來自哪個資料源，**用 raw_items 裡的來源 feed 名**（TLDR 拆出者記 `"TLDR Fintech"` 等 feed 名），供汰換歸因。
     - `decision`：`cover` | `others` | `context` | `dropped`。
     - `reason`：一句繁中，說明為何這樣判，尤其**落選或未當 cover 的原因**。
   - 每個 scope 的 `rejectedSummary`：`{total, eligible, ineligible}`
     （total＝本輪評估的新事件數，eligible＝過資格進 scoredPool 者，ineligible＝其餘）。
3. 先忽略 `_generated_at` 比較 `candidate.json` 與現有 `data.json`。若內容無變化，
   先以 `outcome=no_change` 執行第 13.6 節（直推 wiki），再結束本次任務。
4. 若有變化，將 `candidate.json` 的 `_generated_at` 設為目前 UTC ISO 8601 時間（同步作為 `runAt`）。

## 13.5 兩層把關與修正迴圈

只有第 13 步判定「有變化」時才執行本節。最多重複 3 輪：

1. **第一層（確定性程式）**：
   `python scripts/validate.py candidate.json candidate.meta.json data.json`
   讀取其 JSON 輸出。`ok` 為 false 時，記下所有 `violations`。若 validate.py 以非零狀態結束且沒有可解析的 JSON 輸出（例如程式異常），視為第一層失敗，一併進入修正迴圈；滿 3 輪仍如此則 fail-safe。
2. **第二層（編輯判斷）**：以 Task 工具派 `signal-reviewer` subagent，讓它讀
   `candidate.json`、`candidate.meta.json`、`data.json`、`scripts/raw_items.json` 與本檔規則，
   回傳 `{verdict, issues}`。記下所有 `severity` 為 `block` 的 issue。
3. **判定**：
   - 兩層皆過（validate `ok=true` 且 reviewer 無 block）→ 跳到第 14 節提交。
   - 有任一違規 → 把 validate 的 violations 與 reviewer 的 block issues 串成具體修正清單，
     退回產出端（第 11–13 步）**只修這些問題**後重新產出 `candidate.json` + `candidate.meta.json`，
     回到本節第 1 步，輪數 +1。
4. **防呆**：若本輪的 violations 與上一輪完全相同（同樣的問題沒被修掉），提前中止，不再重試。
5. **fail-safe**：滿 3 輪仍不過，或提前中止 → **保留原 `data.json`、不提交**，先以
   `outcome=fail_safe` 執行第 13.6 節（直推 wiki），再在任務輸出詳列
   每一條未解決的 violation 與 block issue 及其位置。reviewer 的 `minor` issue 只記錄、不阻擋。

## 13.6 寫入 newsroom 選稿日誌

選稿日誌**不進主 repo**，改寫入 GitHub Wiki（`signal-fintech.wiki.git`，一個獨立 repo）。
**每一輪都執行一次**（含 published、no_change、fail_safe），且**每輪僅呼叫一次**（在把關迴圈收斂之後）。

`<outcome>` 依本輪結局擇一：`published`（第 14 步將更新 data.json）、`no_change`（第 13 步第 3 點無變化）、
`fail_safe`（13.5 滿 3 輪仍不過、保留原 data.json）。**三種結局的日誌處理完全相同**（都直推 wiki，無 PR、無合併）。

```bash
# 1) 取得 wiki（獨立 repo，SSH 同一把金鑰、同 host；淺 clone 省容量）
WIKI=/tmp/newsroom-wiki
rm -rf "$WIKI"
git clone --depth 1 git@github.com:laiyenju/signal-fintech.wiki.git "$WIKI" || {
  echo "wiki clone 失敗（多半是 wiki 尚未在網頁建立首頁）；本輪略過日誌、不中斷任務"; WIKI=; }

if [ -n "$WIKI" ]; then
  # 2) 寫入日誌到 wiki（NEWSROOM_DIR 指向 wiki clone；append 當日 json 並重繪 md）
  NEWSROOM_DIR="$WIKI" python scripts/newsroom.py candidate.meta.json candidate.json scripts/raw_items.json <outcome>

  # 3) 直推 wiki（無 PR、無合併回 main）
  git -C "$WIKI" add -A
  git -C "$WIKI" commit -m "newsroom：<today> <outcome> 選稿日誌（$(TZ=Asia/Taipei date +'%Y-%m-%d %H:%M')）"
  git -C "$WIKI" push
fi
```

日誌成對兩檔（`<today>.json` 狀態檔 + `<today>.md` 人看版）都留在 wiki；GitHub Wiki 網頁一天一頁瀏覽。
`data.json` 與日誌自此**完全解耦**：日誌恆進 wiki，`data.json` 只在 published 時進 `main`（第 14 節）。

## 14. 提交、推送與合併

只有第 13.5 節兩層把關皆通過、且 `data.json` 確實有變化時才執行本步。
（本輪 newsroom 日誌已在第 13.6 節以 `outcome=published` 直推 wiki，與本步無關。）
以驗證通過的 `candidate.json` 覆蓋 `data.json`（`cp candidate.json data.json`）。
提交 `data.json`，然後推送當前 `claude/` 分支、建立 PR 並嘗試合併回 `main`：

```bash
git add data.json
git commit -m "自動更新 data.json（$(date -u +'%Y-%m-%d %H:%M UTC')）"
git push -u origin HEAD

gh pr create --base main --head "$(git branch --show-current)" \
  --title "自動更新 data.json（$(date -u +'%Y-%m-%d %H:%M UTC')）" \
  --body "排程任務自動產出，內容依 排程任務指令.md 規則選稿改寫。"

gh pr merge --squash --admin --delete-branch \
  || gh pr merge --squash --auto --delete-branch \
  || echo "自動合併失敗，PR 已建立但需要人工合併"
```

若 PR 已建立但無法自動合併，不要刪除或重做內容；在任務輸出中清楚說明 PR 仍需人工合併。
